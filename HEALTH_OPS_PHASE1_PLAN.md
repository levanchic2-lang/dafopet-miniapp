# 宠物健康运营系统 - 第一阶段开发计划

> 本文件记录 2026-06-30 讨论后已确认的产品路线、边界、开发顺序与第一阶段 MVP 规格。后续开发以此为准。

## 一、总方向

AI+宠物医院的重点不做“AI 问诊噱头”，而是把医院从一次性接诊，升级成持续运营每只宠物健康的系统。

目标链路：

```text
病历详情
→ AI 生成客户版诊后说明
→ 医生编辑确认
→ AI/医生生成复诊计划
→ 医生确认
→ 自动创建回访任务
→ 健康运营入口统一追踪
```

核心目标：

```text
医生少重复解释
客户更听得懂
复诊不丢
慢病有人管
医院能持续运营每只宠物
```

## 二、已确认的五条路线

### 1. AI 预问诊 + 分诊入口

暂缓，不作为第一阶段开发重点。

原因：

```text
很多主人已经会先问 AI、查小红书或问群友，再来问医生。
医院现在更需要体现“医生比 AI 更专业、更懂实际病例、更能落地”。
```

后续可再做，但当前优先级低于医生端增强和复诊运营。

### 2. 客户版诊后说明

第一阶段优先做。

定位：

```text
不是病历，不替代 SOAP。
它是医生确认后，给主人看的诊疗解释和居家护理说明。
```

固定 7 段：

```text
【本次就诊原因】
【医生关注的问题】
【已做检查和结果说明】
【本次用药说明】
【回家护理要点】
【需要尽快联系医院的情况】
【复查建议】
```

第一版功能：

```text
AI 生成
医生编辑
保存草稿
医生确认
复制文本
```

暂不自动发送客户。

### 3. 复诊计划生成

第一阶段优先做。

定位：

```text
把医生临床判断转成可执行任务。
复诊计划不是一段文字，而是结构化任务草稿。
```

任务类型第一版固定：

```text
phone              电话回访
message            微信/小程序消息
photo_check        照片复查
recheck_visit      到店复查
medication_check   用药确认
```

AI 生成草稿后，医生必须确认。确认后才创建 FollowUp。

### 4. 重构自动回访

第一阶段要调整结构，但不废弃现有 FollowUp。

旧逻辑：

```text
surgery +3天
outpatient +7天
beauty +14天
```

新逻辑：

```text
CarePlan 复诊计划
→ FollowUp 执行任务
```

FollowUp 从“简单提醒”升级为“有目的的临床/经营动作”，需要知道：

```text
为什么回访
问什么问题
谁处理
什么情况升级给医生
来源是什么
```

### 5. 慢病与健康运营中心

认同为长期最有价值方向，但第一阶段只做轻量入口。

长期方向包括：

```text
皮肤病
猫泌尿
肾病
心脏病
肥胖
老年宠
牙科
术后恢复
```

第一阶段暂不做慢病标签和套餐转化，只做：

```text
今日待回访
逾期待回访
待确认复诊计划
```

入口第一版放在“医疗”下面，成熟后再考虑升为顶级导航。

## 三、第一阶段 MVP 范围

名称：

```text
诊后说明与复诊计划
```

开发范围：

```text
1. 新增客户版诊后说明 ClientCareSummary
2. 新增复诊计划 CarePlan
3. 扩展 FollowUp 支持 care_plan 来源
4. 病历详情页新增“诊后说明与复诊计划”区域
5. AI 生成诊后说明
6. AI 生成复诊计划 JSON 草稿
7. 医生可编辑、保存、确认
8. 确认复诊计划后创建 FollowUp
9. 医疗下新增“健康运营”入口
10. 健康运营第一版显示今日待回访、逾期待回访、待确认计划
```

暂不做：

```text
自动发送客户
客户小程序查看
慢病标签
套餐转化
复杂统计
自动诊断
AI 自动确认
```

## 四、关键产品原则

```text
AI 只生成草稿
医生必须确认
AI 不新增诊断
AI 不承诺疗效
AI 不建议客户自行调药
AI 不自动创建任务，必须医生确认计划
AI 不可用时，医生仍可手动创建诊后说明和复诊计划
```

客户版诊后说明底部固定风险提示：

```text
以上内容为本次就诊后的护理与复查说明，具体诊断和治疗方案以医生确认的病历记录为准。如症状加重或出现异常，请及时联系医院或到店复查。
```

## 五、数据结构设计

### 1. ClientCareSummary

客户版诊后说明。

建议字段：

```text
id
visit_id
pet_id
customer_id
store
status                 draft / confirmed / archived
source_snapshot_json   生成时引用了哪些病历、处方、检查、报告
ai_draft_text          AI 原始草稿
final_text             医生编辑后的最终文本
doctor_note            医生内部备注，不给客户看
confirmed_by
confirmed_at
created_by
created_at
updated_at
```

设计原则：

```text
一条病历可以有一条当前有效诊后说明
AI 草稿和医生最终稿分开存
确认后不自动发送，只支持复制/保存
source_snapshot_json 用于追溯“AI 当时基于什么生成”
```

### 2. CarePlan

复诊计划。

建议字段：

```text
id
visit_id
pet_id
customer_id
store
summary_id             可为空，关联诊后说明
title
reason
risk_level             low / medium / high
status                 draft / confirmed / active / completed / cancelled
plan_text              给医生看的计划说明
tasks_json             AI 生成或医生编辑后的任务草稿
confirmed_by
confirmed_at
created_by
created_at
updated_at
```

`tasks_json` 示例：

```json
[
  {
    "days_after": 3,
    "due_date": "2026-07-03",
    "task_type": "photo_check",
    "title": "皮肤照片复查",
    "question_text": "请主人拍摄同一部位照片，反馈瘙痒是否减轻。",
    "risk_trigger": "若红肿扩大、破溃、明显疼痛、精神食欲下降，请转医生处理。",
    "priority": "normal"
  }
]
```

### 3. FollowUp 扩展字段

建议新增：

```text
source_type            visit_default / care_plan / chronic / manual
source_id              CarePlan.id 或其他来源 id
reason                 回访原因
question_text          回访时问什么
expected_reply_type    text / photo / phone / visit
risk_trigger           什么情况升级给医生
priority               low / normal / high / urgent
```

第一阶段关系：

```text
CarePlan 草稿阶段：任务存在 tasks_json
医生确认计划后：tasks_json 逐条生成 FollowUp
FollowUp 负责执行、逾期、完成、关闭
```

## 六、页面流程

### 病历详情页

新增区域：

```text
诊后说明与复诊计划
```

顶部操作：

```text
生成诊后说明
生成复诊计划
```

诊后说明状态：

```text
未生成：
  显示“根据本次病历、处方、检查结果生成给主人看的说明”

草稿：
  textarea 可编辑
  按钮：保存草稿 / 确认说明 / 重新生成 / 复制

已确认：
  显示最终文本
  按钮：复制 / 重新编辑（按权限控制）
```

复诊计划状态：

```text
未生成：
  按钮：生成复诊计划 / 手动创建计划

草稿：
  显示计划原因、风险等级、任务列表
  每条任务可编辑：日期、类型、问题、异常条件
  按钮：保存草稿 / 确认并创建回访

已确认：
  显示已创建的 FollowUp 列表
  可跳转到回访详情或健康运营中心
```

### 健康运营入口

入口：

```text
医疗 → 健康运营
```

第一版列表：

```text
今日待回访
逾期待回访
待确认复诊计划
```

每条显示：

```text
宠物名
主人
门店
来源：复诊计划 / 手动 / 默认回访
原因
应处理时间
优先级
最近病历
操作：查看病历 / 完成回访 / 转医生 / 创建预约
```

## 七、AI 生成模板

### 1. 诊后说明系统提示词

```text
你是宠物医院医生的文书助手。你的任务是把医生已经记录的病历、处方、检查结果，整理成给主人看的诊后说明。

你必须遵守：
1. 不新增诊断，不推测系统资料里没有的疾病。
2. 不替医生下最终诊断。
3. 不承诺疗效，不保证恢复时间。
4. 不建议主人自行调整药量、停药或换药。
5. 如果资料不足，要写“需结合医生复查判断”，不要编造。
6. 语言要通俗、具体、温和，像医生对主人解释。
7. 输出只用于医生编辑确认，不能写“我是AI”。

请按固定结构输出：
【本次就诊原因】
【医生关注的问题】
【已做检查和结果说明】
【本次用药说明】
【回家护理要点】
【需要尽快联系医院的情况】
【复查建议】
```

输入资料：

```text
宠物信息
本次病历
处方
检查
收费/治疗
医生额外要求
```

### 2. 复诊计划系统提示词

```text
你是宠物医院医生的复诊计划助手。你的任务是根据医生已确认的病历、处方、检查结果和诊后说明，生成可执行的复诊/回访计划草稿。

你必须遵守：
1. 不新增诊断。
2. 不替医生决定治疗方案。
3. 不让客户自行调整药量。
4. 每个任务都要有明确目的、时间、要问的问题、异常触发条件。
5. 任务数量控制在 1-4 个，避免制造无意义待办。
6. 高风险情况要提示“转医生/建议到店”。
7. 输出必须是合法 JSON，不要输出 Markdown。

任务类型只能使用：
phone, message, photo_check, recheck_visit, medication_check

风险等级只能使用：
low, medium, high

优先级只能使用：
low, normal, high, urgent
```

输出 JSON：

```json
{
  "title": "皮肤问题复诊计划",
  "reason": "本次就诊涉及皮肤红肿和用药观察，需要通过照片和复查判断恢复情况。",
  "risk_level": "medium",
  "plan_text": "建议按 3 天、7 天、14 天进行观察和复查。",
  "tasks": [
    {
      "days_after": 3,
      "task_type": "photo_check",
      "title": "患处照片反馈",
      "question_text": "请主人拍摄同一部位照片，反馈瘙痒、红肿是否减轻。",
      "risk_trigger": "若红肿扩大、破溃、明显疼痛、精神食欲下降，请转医生处理。",
      "priority": "normal"
    }
  ]
}
```

### 3. 默认倾向

```text
术后：
第 1 天精神食欲 / 第 3 天伤口照片 / 第 7 天拆线或复查

皮肤：
第 3 天照片 / 第 7 天复查 / 第 14 天停药或复发观察

猫泌尿：
24 小时排尿确认 / 第 3 天尿检复查 / 第 7 天饮水排尿反馈

消化道：
24-48 小时呕吐腹泻变化 / 第 3 天食欲精神 / 加重立即到店

老年慢病：
根据检查异常安排 14-30 天复查，不要过度频繁

牙科：
术后 1-3 天进食疼痛 / 7 天口腔复查 / 长期洁牙维护提醒
```

## 八、开发顺序

```text
D1：数据库模型和启动迁移
D2：病历详情页诊后说明 UI + 手动保存
D3：AI 生成诊后说明
D4：CarePlan 草稿 UI + 手动编辑任务
D5：AI 生成复诊计划 JSON
D6：确认计划 → 创建 FollowUp
D7：健康运营入口列表
D8：权限、门店过滤、边界测试
```

## 九、验收标准

```text
1. 病历页能手动创建诊后说明
2. AI 能基于病历/处方/检查生成客户可读说明
3. 医生能编辑、保存、确认、复制
4. AI 能生成 1-4 条复诊任务草稿
5. 医生能编辑复诊任务
6. 确认计划后生成 FollowUp
7. 健康运营入口能看到今日、逾期、待确认
8. staff 只能看本门店，superadmin 可筛门店
9. AI 不可用时手动流程仍可用
```
